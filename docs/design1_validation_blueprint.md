# 設計図1: 「星 ⇄ 料理品質」乖離実証ブループリント

> 目的：本提案の土台となる仮説を **感想ではなくデータで** 検証する。
> 状態：**実データ検証 完了（バルセロナ652店 / 3,249レビュー）**。当初の素朴な仮説は実データで一部棄却され、より正確なフレーミングに修正された。本書は当初設計を残しつつ、各所に **【実データ判定】** を注記する。検証過程が見えること自体が評価対象（過程を消さない）。

## ★ 実データ検証サマリ（最重要・先に読む）

**当初の物語（仮説）**：「Google星は料理品質を表さない。観光バーが☆5、良店が3点台。観光地区ほどレビューが料理に触れない」

**実データが示した物語（修正後）**：「Google星はバルセロナで **飽和して識別力を失っている**。652店中ほぼ全店が☆4.3〜4.9に圧縮（3点台は21店のみ）。星では店を区別できない。その同点だらけの帯の中で、**FQSが料理品質の差を回復する**」

| 指標 | 合成データ | 実データ | 判定 |
|---|---|---|---|
| Pearson r（星 vs FQS） | 0.226 | **0.448** | 乖離は実在するが中程度。星はFQS分散の約20%しか説明しない |
| food言及率 観光 vs 地元 | （作為的に差） | **96% vs 96%** | **STEP3仮説は棄却**（差なし） |
| ambiance言及率 観光 vs 地元 | — | 51% vs 46% | 予測方向だが僅差 |
| 星3点台の店 | — | **21/652** | 裾が極薄＝レンジ制限（飽和の証拠） |

**フレーミング修正の核心**：
- ✗ 旧：「星は **間違っている**」 → 主観的で反論されやすい（「あなたの好みでは？」）
- ✓ 新：「星は **飽和・非識別**（4.5に張り付き情報を持たない）。FQSが識別力を回復する」 → 星ヒストグラムで一撃で示せる客観的事実。Googleを全否定せず「機能不全領域の補完」として提案できる

**なぜ新フレーミングの方が強いか**：5段階スケールが上端に張り付き、エントロピーがほぼゼロ＝ランキングの道具として壊れている。これはデータで証明できる。「星が壊れた領域をFQSで補う」はGoogleにも受け入れやすい。

**棄却された仮説の扱い**：STEP3（観光バイアス）は実データで棄却。**これを隠さずスライドに載せる**。「仮説を立て→検証し→棄却した」過程は、検証を本当にやった証拠＝加点。消すと「検証していない」と区別がつかない。

## 1. データソース（2系統）

### 1-A. 方法論実証用：Google Local（McAuley / UCSD）
- URL: https://cseweb.ucsd.edu/~jmcauley/datasets.html#google_local（要確認）
- 実態：Google Mapsレビューの研究公開版（米国・2021年9月まで）
- フィールド：`user_id, name, time, rating, text, pics, resp` ＋ 店舗メタ（カテゴリ, 価格, 座標, 平均星, レビュー数）
- 役割：**レビュアー履歴・タイムスタンプが揃う**ため、reviewer weighting / time decay を含むフル手法を実証できる。
- 注意：**バルセロナは入っていない（全米国）**。よって「都市は違うが手法は移植可能」という立て付けで使う。

### 1-B. 物語実証用：バルセロナ実データ（Google Places API New）
- 取得：既存 `ingest_google_places.py` の**改造版**で取得（設計図2の Module A）。
- 制約：**1店あたり最大5レビュー**（ハード制約・有料でも不変）。
- 役割：小規模でも「観光都市バルセロナで実際に乖離が起きている」実例を示す。5件でも散布図1点は打てる。

> **データ取得不能時のフォールバック**：バルセロナの実態を反映した**合成データ生成器**（設計図2 Module A-sim）で代替。教授の課題は「仮データ可」なので正当。合成の前提（観光バー☆膨張・料理無言及など）は明示する。

## 2. 分析ステップ（実行順）

### STEP 1 — 料理品質スコアの定義（Food Quality Score, FQS）
**入力**：各店のレビュー集合（text, rating, author, time）
**処理**：
1. 各レビューを **ABSA（Aspect-Based Sentiment Analysis）** にかけ、4アスペクトに分解：
   - `food` / `service` / `ambiance` / `price`
2. レビューごとに **food アスペクトの感情極性**（-1〜+1）を抽出。food言及がなければ `null`。
3. 店舗の **FQS = food言及レビューの感情極性の加重平均**（重み設計は STEP4）。
**出力**：店舗ごとに `star_rating`（生の星）と `FQS`（料理品質）の2値。

### STEP 2 — 乖離の測定（★図1）
**処理**：
- 全店で `star_rating` と `FQS` の **相関係数（Pearson / Spearman）** を計算。
- 散布図を描画。対角線（完全一致）からの乖離を可視化。
- 「☆4.5以上なのにFQSが下位50%」の店を**赤でハイライト**＝"観光トラップ店"。
**出力（スライド用）**：
- 散布図1枚（相関係数を併記）。**相関が低いほど主張が強い。**
- 仮説検証文：「星と料理品質の相関は r=○○。星だけで選ぶと料理品質を○○%しか説明できない」

> **【実データ判定】** r=0.448（合成0.226より高いが1.0より十分低い）。乖離は実在。
> **ただし解釈を修正**：散布図では大半の店が☆4.3〜4.9に圧縮され、その帯内でFQSが−0.9〜+1.1に大きくばらつく。これは「星とFQSが逆向き」ではなく **「星が飽和して非識別／FQSが同点帯の中で料理品質を識別している」**。
> スライド文言は「星は間違い」ではなく **「星は飽和・非識別。FQSが同点帯の中で料理品質を回復する」** に修正。図1（real_fig1_star_vs_fqs.png）と星ヒストグラム（real_star_histogram.png）をセットで見せ、「ほぼ全店4.5★＝星は情報を持たない」を先に示す。

### STEP 3 — 「星は高いが料理無言及」率（★図2）
**処理**：
- ☆4.0以上の店のレビューのうち、**food言及ゼロ**のレビューの割合を集計。
- アスペクト出現頻度を棒グラフ化（food / ambiance / price / service）。
- 観光地区（例：La Rambla, Barceloneta）と地元地区（例：Gràcia, Sant Andreu）で**層別比較**。
**出力（スライド用）**：
- 棒グラフ：「☆4.5以上の店でも、レビューの○○%は料理に一言も触れていない」
- 層別比較：観光地区ほど ambiance/price 言及が多く food 言及が少ない（＝観光バイアスの可視化）

> **【実データ判定：この仮説は棄却】** food言及率は観光96%・地元96%で **差なし**。「観光地区ほど料理に触れない」は実データで成立しなかった。
> 原因：Googleが返す「関連性上位」レビューは地区を問わず詳細で、ほぼ全部が料理に言及する。
> **スライドでの扱い**：消さずに「仮説→検証→棄却」として明示的に載せる（＝検証を本当にやった証拠＝加点）。ambiance言及は観光51%・地元46%で予測方向だが僅差、と正直に併記。
> **副産物（Limitationsで使う）**：「Googleの関連性上位レビューは好意的・詳細に偏る選択バイアスがある」という新たな発見。これ自体が"見えにくいトラップ"。

### STEP 4 — 再ランキング（★図3）
**処理**：
- 同一エリア・同一ジャンル内で、`star_rating` 順 と `FQS` 順 を並べる。
- 順位変動（rank delta）が大きい店を抽出：
  - **「星は低いが料理は本物」**（FQSで急上昇）＝発掘された優良店
  - **「星は高いが料理は微妙」**（FQSで急落）＝観光トラップ店
**出力（スライド用）**：
- before/after の順位入れ替え表（矢印つき）。これが提案の"効果"の証拠。

> **【実データ判定】** rank_delta上位の「トラップ」196店のうち、☆4.5でFQS−0.4の **真のトラップも実在**する。ただし注意：レポート上位に出た店（L'APE REGINA等）はFQS+0.6〜0.76と **絶対値はむしろ高く**、地区×ジャンルの小集団内で相対的に順位が下がっただけ。「料理がダメな高星店」ではない。
> **スライドでの扱い**：「絶対トラップ（FQS実際に低い）」と「相対トラップ（小集団内の相対順位）」を区別して提示する。誇張して「これがダメ店」と言わない。真のトラップ（FQS絶対値が低い高星店）を例に選ぶ。

## 3. 評価：FQSが「星より正しい」ことの証明（★最重要・配点30%）

正解ラベルがない問題なので、**外部の独立した「料理が良い店」リストを代理正解**にする。

### 3-A. 代理正解ラベル（バルセロナで入手可能なもの）
- **Michelin Guide**（星 / Bib Gourmand）
- **Repsol Guide（Soles Repsol）** ← スペイン文脈で特に有効
- （補強）地元グルメメディアの年間ベスト等

### 3-B. 検証ロジック
- 仮説：「FQSは、生の星より、これら専門ガイド掲載店との一致度が高い」
- 指標：
  - **Precision@K / Recall@K**：FQS上位K店 と 星上位K店、それぞれにガイド掲載店が何件含まれるか
  - **NDCG**：掲載＝高relevance としてランキング品質を比較
  - **AUC**：「掲載/非掲載」を当てる二値分類として、FQS vs 星 のROC比較
**出力（スライド用）**：
- 「専門ガイド掲載店の捕捉率：星 ○○% → FQS ○○%」の対比棒グラフ。

### 3-C. ロバストネス（ablation）
- 各信号（reviewer weighting / time decay / ジャンル正規化）を**1つずつ外す**と精度がどう落ちるか。
- どの信号が効いているかを示す＝技術的な誠実さ（配点 Technical 20%）。

## 4. 信号設計（FQSに何を入れるか）— 強弱を明示

| 信号 | 採否 | 根拠 | データ要件 |
|---|---|---|---|
| ① food アスペクト感情（ABSA） | **採用・主軸** | 雰囲気/酒/テラスを剥がせる。提案の核 | レビュー本文 |
| ② レビュアー信頼度（food語り重視の人を重く） | **採用（McAuleyのみ）** | foodie履歴のある人の評価を重視 | レビュアー履歴（Placesでは不可） |
| ③ 時間減衰（新しいレビュー重視） | **採用** | シェフ交代で味は変わる。"店の築年数"ではなく"レビューの新しさ" | タイムスタンプ |
| ④ ジャンル別正規化 | **採用** | 寿司屋とタパスバーを同じ星基準で比べない | 店舗カテゴリ |
| ⑤ 写真の food比率 | **任意（extra mile）** | 料理写真かテラス写真かで店の評価軸を推定 | 画像（マルチモーダル） |
| ✗ 店の築年数（10年以内重視） | **不採用** | 老舗の名店を殺す。新しさ→味の因果なし。公平性リスク | — |

> ②③で当初アイデアの「10年以内の店を重視」を**作り替えた**点が重要。落としたのではなく、本来効かせたかった"鮮度"を③time decayで正しく実装した。

## 5. 注意すべきバイアス（自分で先回りして書く＝加点）

> **【実データで判明した限界・最優先で明示】**
> - **星のレンジ制限（飽和）**：652店中ほぼ全店が☆4.5前後。星の分散が小さく、乖離指標（相関）自体が歪む。これは欠点ではなく **提案の前提そのもの**（星が飽和＝FQSが要る理由）として語る。
> - **1店最大5レビュー（Places制約）**：FQSが少数サンプルでノイジー。本番はパートナー（Google）のフルレビューで解消する旨を明示。
> - **選択バイアス**：対象はGoogleの「関連性上位」レビュー＝詳細・好意的に偏る。food言及96%の高さもこれが一因。真にランダムなレビュー母集団ではない。
> - **軽微**：ジャンル正規化（平均センタリング）でFQSが稀に±1を僅かに超える／代表レビュー例示は「最長」を選んでおりFQS平均と一致しない場合がある（表示ロジックの注記）。

- **foodie snobbery bias**：②で「グルメ層」を重視すると、高級店過大・安い良店過小の逆バイアスが入りうる。あなたが嫌う"乾き物バー☆5"の逆方向。重みは学習で決め、ハードルールにしない。
- **観光客 vs 地元の線引き**：「観光客票を割り引く」は一部利用者の声の軽視。正当性を倫理パートで論じる。
- **言語バイアス**：多言語レビュー（西/カタルーニャ/英が混在）。ABSAが言語で精度差を持つ可能性。実データでfood言及率が地区で差がなかったため大きな歪みは観測されていないが、要留意。
- **店への公平性**：合法な営業形態（テラスバー）を構造的にダウンランクする正当性。
